# MnemonicFlow

Приложение для интервального повторения flashcards с адаптивной системой запоминания...

### Основные возможности

- **Интервальное повторение** — алгоритм SM-2 для эффективного запоминания
- **Типы карточек** — обычные, с пропусками, множественный выбор
- **Мультимедиа** — поддержка изображений и аудио для карточек
- **Комментарии** — обсуждение карточек в режиме изучения
- **Статистика** — отслеживание прогресса и серий изучений
- **Импорт из Anki** — загрузка колод из .apkg файлов
- **PWA** — работает оффлайн, доступна для установки

## 🏗️ Архитектура фронтенда (FSD)

Фронтенд построен по методологии **Feature-Sliced Design** — слои изолированы и импортируются только сверху вниз:

```
┌─────────────────────────────────────────┐
│  app/      — глобальная инфраструктура  │
├─────────────────────────────────────────┤
│  pages/    — страницы приложения       │
├─────────────────────────────────────────┤
│  widgets/  — крупные reusable блоки    │
├─────────────────────────────────────────┤
│  features/ — пользовательские действия  │
├─────────────────────────────────────────┤
│  entities/ — доменные сущности         │
├─────────────────────────────────────────┤
│  shared/   — общие инструменты         │
└─────────────────────────────────────────┘
```

**Правило импортов:** слой может импортировать только из слоёв **строго ниже**.

Примеры:
- ✅ `pages/` → `features/` → `entities/` → `shared/`
- ❌ `features/` → `features/` (запрещено)
- ❌ `entities/` → `pages/` (запрещено)

## 👤 Аутентификация и профиль пользователя

### Регистрация и онбординг

При регистрации нового пользователя:
1. Email должен быть подтверждён через письмо
2. После первого входа пользователь попадает на онбординг для ввода имени
3. Приложение блокирует доступ к функционалу пока имя не будет установлено

### Управление профилем

- **Редактирование имени:** доступно через профиль (кнопка с иконкой карандаша)
- **API эндпоинты:**
  - `PATCH /api/auth/me/username` — обновление имени пользователя
  - `GET /api/auth/me` — получение данных текущего пользователя

## 📚 Колоды и карточки

### Просмотр карточек колоды с пагинацией

Приложение поддерживает пагинацию карточек при просмотре колоды для оптимизации производительности.

**Особенности:**
- По умолчанию отображается 15 карточек на странице
- Навигация между страницами с отображением текущей страницы и общего количества
- Информация о диапазоне карточек (например, "Карточки 1–15 из 42")
- Консистентная сортировка по дате создания карточки

**API эндпоинт:**

```http
GET /api/decks/{deck_id}/cards?page=1&per_page=15
```

**Query параметры:**
- `page` (int, по умолчанию 1) — номер страницы (минимум 1)
- `per_page` (int, по умолчанию 15) — количество карточек на странице (от 1 до 50)

**Response:**
```json
{
  "cards": [
    {
      "card_id": "uuid",
      "title": "Card Title",
      "type": "flashcard",
      "levels": [
        {
          "level_index": 0,
          "content": {},
          "question_image_urls": [],
          "answer_image_urls": [],
          "question_audio_urls": [],
          "answer_audio_urls": []
        }
      ]
    }
  ],
  "total": 42,
  "page": 1,
  "per_page": 15,
  "total_pages": 3
}
```

## 📊 Дашборд и статистика

### Главный экран

Главный экран (Home) отображает:
- **Статистика за сегодня:** количество изученных карточек, время обучения, текущая серия
- **Диаграмма сложности:** пончиковая диаграмма (donut chart) распределения карточек по категориям:
  - 🟢 **Легко** (difficulty 1-3) — зелёный
  - 🟡 **Средне** (difficulty 4-6) — янтарный
  - 🔴 **Сложно** (difficulty 7-10) — красный
- **Кнопка "Начать обучение":** зафиксирована при прокрутке, запускает сессию повторения

При клике на центр диаграммы происходит переход на вкладку статистики.

### Страница статистики

Страница статистики предоставляет расширенную аналитику прогресса обучения:

**Общая статистика:**
- Общее время обучения (с форматированием "Xh Ym" или "X days Yh")
- Средняя длительность сессии
- Скорость обучения (новых карточек в день)
- Распределение оценок (Again/Hard/Good/Easy) с круговой диаграммой
- Средняя оценка (1-4 шкала)

**Тепловая карта активности:**
- GitHub-style календарь активности за последние 365 дней
- Цветовая индикация интенсивности изучений
- Tooltip с деталями при наведении

**Прогресс по колодам:**
- Список всех колод с цветовыми индикаторами
- Прогресс-бар (выученные / изучаемые / новые)
- Статистика по каждой колоде (количество повторений, время изучения)
- Порог выученности: 30+ дней стабильности

**Графики активности:**
- Линейный график тренда активности
- Столбчатый график времени изучения
- Селектор периода: день / неделя / месяц
- Данные за последние 30 дней (настраиваемо)

### API эндпоинты статистики

| Эндпоинт | Описание |
|----------|----------|
| `GET /api/stats/dashboard` | Основная статистика для дашборда |
| `GET /api/stats/difficulty-distribution` | Распределение карточек по сложности |
| `GET /api/stats/general` | Общая статистика (время, сессии, рейтинги) |
| `GET /api/stats/activity-heatmap?days=365` | Данные для тепловой карты активности |
| `GET /api/stats/deck-progress` | Прогресс по каждой колоде |
| `GET /api/stats/activity-chart?period=day|week|month&days=30` | Данные для графиков активности |

## 📥 Импорт из Anki

Приложение поддерживает импорт колод из Anki (`.apkg` файлы).

### Формат .apkg

Anki `.apkg` файл — это ZIP-архив, содержащий:
- SQLite базу данных (`collection.anki2` или `collection.anki21`)
- Файл `media` с маппингом медиа-файлов
- Медиа-файлы (изображения, аудио)

### Как импортировать

1. На экране колод нажмите "Добавить колоду"
2. Выберите "Импорт из Anki"
3. Выберите `.apkg` файл
4. Колоды и карточки будут созданы автоматически

### Ограничения

- **Максимальный размер файла:** 50MB
- **Поддерживаемые форматы:** `.apkg`
- **Конвертация контента:** HTML из Anki преобразуется в текст
- **Медиа:** В текущей версии медиа-файлы не импортируются (планируется)

### API эндпоинт импорта

| Эндпоинт | Описание | Auth |
|----------|----------|------|
| `POST /api/decks/import-anki` | Импорт колоды из .apkg файла | ✅ |

**Request:** `multipart/form-data` с полем `file` (`.apkg` файл)

**Response:**
```json
{
  "deck_id": "uuid",
  "title": "Deck Name",
  "cards_created": 42,
  "warnings": []
}
```

## 🏷️ Версионирование

Проект использует **Semantic Versioning** (формат `MAJOR.MINOR.PATCH`):

- `MAJOR` — Breaking changes (обратная совместимость нарушена)
- `MINOR` — Новые функции (обратная совместимость сохранена)
- `PATCH` — Bugfixes (обратная совместимость сохранена)

### Получение версии

**Backend:**
```bash
# Через API
curl http://localhost:8000/version
# {"version": "0.0.99"}

# В коде
from app.core.version import get_version
print(get_version())  # "0.0.99"
```

**Frontend:**
```typescript
import { APP_VERSION } from '@/shared/lib/version';

console.log(APP_VERSION);  // "0.0.99"
```

### Обновление версии

Версия хранится в файле `VERSION` в корне проекта. Для обновления версии:

1. Откройте файл `VERSION`
2. Измените номер версии (например, `0.0.99` → `0.1.0`)
3. Пересоберите приложение:
   ```bash
   docker compose -f compose.dev.yml up -d --build frontend
   docker compose -f compose.dev.yml up -d --build backend
   ```

### Автоматический процесс релиза

Проект использует автоматизированный CI/CD пайплайн для создания релизов:

**1. Подготовка release ветки:**

Скрипт `.github/workflows/merge-to-release.sh` (запускается вручную из develop):
```bash
# Переключиться на develop
git checkout develop

# Запустить скрипт
.github/workflows/merge-to-release.sh
```

**Что делает скрипт:**
- Читает версию из `VERSION` (например, `0.0.6-dev`)
- Убирает суффикс `-dev` → `0.0.6`
- Создаёт временную ветку `release_v0.0.6`
- Обновляет `VERSION` до `0.0.6` (без `-dev`)
- Коммитит `release: update to v0.0.6`
- Мержит временную ветку в `release_v0.0.x`
- Удаляет временную ветку

**2. Автоматический релиз (`.github/workflows/release.yml`):**

При пуше в любую release ветку (`release_v*.x`):
1. Валидирует VERSION файл
2. Создаёт git tag `vX.Y.Z`
3. Создаёт GitHub Release с changelog
4. Мержит изменения в `main`
5. Bump версии в `develop` (например, `0.0.6` → `0.0.7-dev`)

**3. Версионирование в develop:**

Ветка `develop` всегда должна иметь суффикс `-dev`:
- Перед релизом: `0.0.6-dev` в `develop` → `0.0.6` в `release_v0.0.x`
- После релиза: автоматически bump до `0.0.7-dev` в `develop`

**Ручное создание релиза (альтернатива):**

1. Обновите версию в файле `VERSION`
2. Закоммитьте изменения: `git commit -am "chore: bump version to X.Y.Z"`
3. Создайте git tag: `git tag vX.Y.Z`
4. Запушьте тег: `git push origin vX.Y.Z`
5. Создайте Release на GitHub на основе тега

## 🔧 CI/CD

Автоматический CI запускается для всех PR в ветки `main` и `develop` через единый пайплайн `ci.yml`:

1. **validate-commits** — Валидация формата VERSION и Conventional Commits
2. **code-style** — Проверка код стиля (pre-commit hooks)
3. **build-all** — Сборка Docker образов с кешированием
4. **test-all** — Запуск тестов (pytest + vitest)

**Публикация образов** (`push-images.yml`) — Отдельный workflow, запускается:
- При пуше в ветки `main` или `develop`
- Вручную через `workflow_dispatch`

**Docker Registry:** `ghcr.io/<username>/mnemonic_flow/` (замените `<username>` на вашего владельца репозитория)
